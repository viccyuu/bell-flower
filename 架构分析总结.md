# Chujiu Project 架构分析总结

**分析时间**: 2026-03-17  
**项目版本**: 2.0.0 (FastAPI)

---

## 📊 核心发现

### 1. 架构类型
**单体应用（Monolithic）带逻辑模块划分**
- 单一代码库、单一部署单元
- 共享 SQLite 数据库
- 按业务能力划分 20 个逻辑模块

### 2. 模块统计

| 类型 | 数量 | 复杂度 |
|------|------|--------|
| 业务模块 | 7 | 中 |
| 支撑模块 | 4 | 高 |
| 基础设施 | 3 | 低 |
| 公共模块 | 3 | 低 |
| 外部适配 | 3 | 中 |
| **总计** | **20** | **中** |

### 3. 技术栈

| 层次 | 技术 |
|------|------|
| Web 框架 | FastAPI 0.104.1 |
| 数据库 | SQLAlchemy 2.0 + aiosqlite |
| 数据验证 | Pydantic 2.5 |
| LLM 适配 | litellm |
| 测试 | pytest + playwright |

---

## 🎯 关键架构特点

### 1. 分层清晰
```
API 层 (Routes)
    │
    ▼
业务层 (Task/Upload/Plugin...)
    │
    ▼
支撑层 (Scheduler/Executor/Resolver)
    │
    ▼
基础设施层 (Database/Repository/Config)
    │
    ▼
数据存储 (SQLite + 文件系统)
```

### 2. 核心调度器
**scheduler 模块**是系统核心：
- DAG 解析与验证
- 节点调度与执行
- 状态管理与重试
- 依赖 4 个执行器（Python/LLM/File/Todolist）

### 3. 插件化设计
- 自动扫描 `app/plugins/` 目录
- 动态导入注册
- DAG 配置驱动
- 支持启用/禁用

### 4. 异步优先
- FastAPI 异步框架
- SQLAlchemy Async ORM
- aiosqlite 异步驱动
- 后台任务执行

---

## ⚠️ 风险点

### 1. 高耦合模块
**scheduler 模块**耦合度最高：
- 依赖 repository、executor、resolver、log_service
- 建议通过接口抽象降低耦合

### 2. 共享数据库
- 所有模块共享 SQLite
- 存在锁竞争风险
- 建议生产环境迁移到 PostgreSQL/MySQL

### 3. 边界模糊
- plugins 模块既是业务又是支撑
- 建议拆分为 plugin_management 和 plugin_registry

### 4. 缺失模块
- todolist 模块代码证据不足
- agent 模块使用情况不明
